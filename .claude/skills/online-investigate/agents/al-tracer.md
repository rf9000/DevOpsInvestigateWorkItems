## Field Name Transformation

AL Payment Entry field names are transformed to JSON keys by `TextHelper.RemovePaymentEntryPrefixesAndIllegalCharacters()`:

1. Remove prefixes (in order): `"Debtor "`, `"Creditor "`, `"Crdtr "`, `"Bank "`, `"CdtrRefInf "`, `"RfrdDocInf "`, `"Tx "`
2. Remove illegal chars: spaces, hyphens, parentheses, slashes, periods

Examples:
- `"Creditor Bank Name"` → remove "Creditor " → `"Bank Name"` → remove "Bank " → `"Name"` → remove spaces → `"Name"`
- Wait, that's wrong. The prefixes are removed sequentially with early exit. Let me correct:
  - `"Creditor Bank Name"` → remove "Creditor " → `"Bank Name"` → then try "Bank " → `"Name"` → `"Name"`
  - But actually in the Creditor Bank context, the field is listed in the bank object, so context matters.

**When searching for a JSON key, reverse the transformation:**
- Take the JSON key (e.g., `"BankName"`)
- Insert likely prefix + spaces: try `"Creditor Bank Name"`, `"Debtor Bank Name"`, `"Bank Name"`
- Search Payment Entry table for matching field names

## Strategy

### Step 1: Determine if SEARCH_TERM is a JSON key or AL field name

- If it contains spaces or known prefixes (Debtor, Creditor, etc.) → it's an AL field name
- If it's PascalCase without spaces → it's likely a JSON key
- Search both ways if unclear

### Step 2: Find the Field in Payment Entry

Search `PaymentEntry.Table.al` for the field name or reversed-transformed name:
- Grep for the SEARCH_TERM in the table definition
- Note the field number, type, and exact name

### Step 3: Trace the Export Chain (outbound)

1. Search `InhouseFileStructure.Codeunit.al` for the field number (e.g., `FieldNo("Creditor Bank Name")`)
2. Find which section/procedure adds it to a FieldList
3. This tells you which JSON object it belongs to (root, PaymentInfo, TransactionInfo, CreditorBank, etc.)
4. The JSON key is generated by `RemovePaymentEntryPrefixesAndIllegalCharacters(FieldName)`

### Step 4: Trace the Value Source (where the data comes from)

1. Search for where the Payment Entry field is populated:
   - Grep for `.FieldNo("FieldName")` or `"FieldName" :=` across the codebase
   - Look in `CreateCreditInhouse.Codeunit.al`, `CreateDebitInhouse.Codeunit.al`
   - Look in Payment Journal, Gen Journal, vendor/customer tables
2. Follow the chain back to the originating table/page

### Step 5: Trace the Import Chain (inbound, if applicable)

1. Search for the JSON key or field name in import-related codeunits
2. Look in `csv-import/` and `import/` modules
3. Check for any field mapping tables or configuration

## Important Notes

- The Payment Entry table is TEMPORARY - it's populated at export time from Payment Journal lines, vendor/customer data, bank accounts, etc.
- Multiple fields may contribute to one JSON key (e.g., address lines)
- Some fields are computed/derived during export (not directly from a single source field)
- The InhouseFileStructure organizes fields into nested JSON objects: Root → PaymentInformation[] → TransactionInformation[] → CreditorBank, DebtorBank, etc.
